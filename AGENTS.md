## ⚙️ The Execution Lifecycle

You must strictly adhere to the following phase-gate process for **every single task**. Do not skip steps.

### Phase 1: Pre-Flight Synchronization
1. **Context Initialization:** Identify the domains your task touches.
2. **Pre-Task Doc Update:** Before writing any code, update the respective `docs/product-specs/`, `docs/exec-plans/active/`, or `ARCHITECTURE.md` files to reflect the incoming task's requirements and intended design.

### Phase 2: Test-Driven Development (TDD) Protocol
1. **Test Plan:** Draft a clear test plan in your active execution plan document.
2. **Generate Tests:** Write the failing test cases for the feature or fix.
3. **Execute Development:** Write the application code required to pass the newly generated tests. Loop locally until all agent reviewers, linters, and tests are green. Ensure strict architectural boundaries are maintained (e.g., code only depends "forward" through predefined layers).

### Phase 3: Post-Flight Teardown
1. **Post-Task Doc Update:** If your implementation altered data shapes, architecture, or configurations, you must immediately update the relevant documentation (e.g., `docs/generated/db-schema.md`, `DESIGN.md`) to reflect the new reality.
2. **Commit & Push:** Once tests pass and documentation is synced, commit your changes to the active branch. Provide a concise, highly descriptive commit message detailing the task completed.
